1. System Architecture

End-to-End Flow

                             │  REST endpoints     │      │  Interactive viewer │
                             └────────────────────┘      └─────────────────────┘

Key Design Decisions

Decision Rationale
Datalab Marker for extraction Handles complex PDF layout, tables, equations, images better than pdfplumber/pymupdf alone
Single JSON file storage Simple, portable, no database dependency for the prototype
Streamlit viewer Rapid iteration without a separate frontend build step
FastAPI alongside Streamlit Enables future React/headless integrations without rewriting
KaTeX for math rendering Browser-native, no server-side LaTeX processing needed

2. Prerequisites & Installation

System Requirements

  • Python 3.10 or higher
  • pip (comes with Python)
  • Internet access for the Datalab API call (first run only per PDF)
  • ~500 MB disk space for a typical 150-page PDF (raw cache + structured output)

Step-by-Step Installation

1. Get the code

git clone <repository-url>
cd buildingCodeWebProject

2. Create a virtual environment (strongly recommended)

# Windows
python -m venv .venv
.venv\Scripts\activate

# macOS / Linux
python3 -m venv .venv
source .venv/bin/activate

3. Install Python dependencies

pip install -r requirements.txt

The dependencies are:

Package Purpose
requests HTTP calls to Datalab API
anthropic Claude API SDK
pdfplumber PDF reading fallback
pymupdf PDF reading fallback
fastapi REST API framework
uvicorn ASGI server for FastAPI
python-multipart File upload support for FastAPI
python-dotenv Load .env file
streamlit Web viewer framework
pandas Data utilities

3. Configuration

API Keys

Create a .env file in the project root directory:

DATALAB_API_KEY=dl_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

Getting a Datalab key:
1. Create an account at https://www.datalab.to
2. Navigate to API Keys in your dashboard
3. Generate a new key

Getting an Anthropic key:
1. Create an account at https://console.anthropic.com
2. Navigate to API Keys
3. Generate a new key (only needed if using --ai flag)

Environment Validation

Both clients validate their keys on startup and print a clear error if missing:

[ERROR] DATALAB_API_KEY is not set.
Steps to fix:
  1. Open your .env file
  2. Set DATALAB_API_KEY=your_actual_key
  3. Get a key at https://www.datalab.to/app/keys

4. Running the Pipeline

Basic Usage

python main.py path/to/your_document.pdf

Pipeline Steps

The pipeline runs four sequential steps:

[Step 1/4] Extracting PDF...
  → Submits PDF to Datalab API (or loads from cache)
  → Saves/loads: storage/raw_{pdf_stem}.json

[Step 2/4] Parsing document structure...
  → Builds Chapter → Section → Clause hierarchy
  → Extracts tables, figures, equations
  → Saves figures to: storage/figures/

[Step 3/4] Linking internal references...
  → Resolves cross-references (Sentence X, Table X, etc.)
  → Resolves appendix note refs (See Note A-X.X.X.(N).)
  → Prints resolution statistics

[Step 4/4] Saving / AI enhancement...
  → Writes: storage/output/structured_document.json

Command-Line Flags

# Re-call Datalab API (skip cache) — use when you have a new version of the PDF
python main.py doc.pdf --force-extract

# Enable Claude AI enhancement (adds semantic column labels to tables)
python main.py doc.pdf --ai

# Both flags
python main.py doc.pdf --force-extract --ai

Caching

On the first run, the Datalab API response is saved to storage/raw_{pdf_stem}.json. Every subsequent run loads from this cache instead of calling the API. This means:

  • Parser/viewer changes can be tested at zero API cost
  • Only use --force-extract when the PDF itself changes

Expected Output

============================================================
  EXTRACTION COMPLETE
============================================================
  Document:    Part 4 Structural Design
  Pages:       146
  Chapters:    1
  Sections:    31
  Clauses:     202
  Tables:      28
  References:  366 found, 93.2% resolved
  Output:      storage\output\structured_document.json
============================================================

5. Data Model

The structured document is a nested JSON tree:

Document
  ├── title: str
  ├── source_pdf: str
  ├── total_pages: int
  ├── extracted_at: str  (ISO timestamp)
  ├── _stats: { reference counts }
  └── chapters: [Chapter]
        └── Chapter
              ├── id: "CH-4"
              ├── number: "4"
              ├── title: "Structural Design"
              └── sections: [Section]
                    └── Section
                          ├── id: "SEC-4-1"
                          ├── number: "4.1"
                          ├── title: "Loads"
                          └── clauses: [Clause]
                                └── Clause
                                      ├── id: "CL-4-1-6-5"
                                      ├── number: "4.1.6.5"
                                      ├── title: "..."
                                      ├── page_span: [int, ...]
                                      ├── content: [ContentItem]
                                      ├── tables: [Table]
                                      ├── figures: [Figure]
                                      ├── equations: [Equation]
                                      ├── references: [Reference]
                                      └── note_refs: [NoteRef]

ContentItem types

Every piece of clause body content is stored as an ordered content[] array to preserve PDF reading order:

type Fields Description
text value: str Plain paragraph or inline-math text ($...$ notation)
sub_clause marker: str, value: str Labelled sub-item, e.g. (a), i., a)
equation latex: str, id: str Display equation, one per <math> block
figure figure_id: str, image_path: str, caption: str Embedded image
table table_id: str, value: str Reference to a table in tables[]

Table structure

{
  "id": "TBL-14",
  "caption": "Table 4.1.6.2.-B ...",
  "headers": ["Value of $l_c C_w^2$", "Value of $C_w$ / 1.0 / Value of $C_b$"],
  "rows": [["70", "0.80"], ["80", "0.82"]],
  "page": 42,
  "column_semantics": []
}

Reference structure

{
  "text": "Article 4.1.6.5.",
  "kind": "Article",
  "target_id": "CL-4-1-6-5",
  "resolved": true
}

Note reference structure

{
  "raw": "(See Note A-4.1.6.16.(6).)",
  "note_ref": "A-4.1.6.16.(6).",
  "target_ids": ["CL-AUTO-49"],
  "resolved": true
}

ID Naming Conventions

Node type Pattern Example
Chapter CH-{n} CH-4
Section SEC-{n}-{m}[-{k}…] SEC-4-1, SEC-4-1-6
Clause CL-{n}-{m}-{k}[-{j}…] CL-4-1-6-5
Unnumbered clause CL-AUTO-{n} CL-AUTO-39
Table TBL-{n} TBL-14
Figure FIG-{n} FIG-3
Equation EQ-{n} EQ-2

Dots in numbers are replaced with hyphens: 4.1.6.5 → CL-4-1-6-5.

6. Module Reference

6.1 Ingestion — datalab_client.py

Handles PDF submission and result retrieval from the Datalab Marker API.

Key parameters sent to Datalab:

{
    "output_format": "json",
    "use_llm": "true",       # enables LLM-enhanced extraction
    "extract_images": "false"
}

Polling: 5-second interval, 300-second timeout. Raises TimeoutError if the job does not complete.

Cache path: storage/raw_{pdf_stem}.json — derived from the PDF filename (spaces → underscores).

6.2 Parser — structure_parser.py

Converts the Datalab JSON into the structured document tree.

Entry point:

from parser.structure_parser import parse_datalab_output

document_dict = parse_datalab_output(
    datalab_result,
    source_pdf="my_doc.pdf",
    figures_dir="storage/figures"
)

Heading-to-hierarchy mapping:

Heading level Rule
h1 New Chapter
h2 New Section (if matches N.N pattern)
h3 New Section (if matches N.N.N); else label clause
h4 New Clause (checks 4-part number first, then 3-part)
h5 Notes / Appendix entries → CL-AUTO-N
h6 Bold text item in current clause; or new unnumbered clause

Text block auto-promotion: Plain text blocks whose first line matches a structural number pattern are promoted to Section or Clause automatically, with duplicate-ID guards.

Math handling:
- <math> tags in body text → $...$ inline notation (rendered by Streamlit/KaTeX)
- <math> tags in Equation blocks → one EQ-N ContentItem per tag

Table merging: Cross-page (continued) table fragments are detected and merged into the base table after parsing. Cross-page rowspan carry is applied via a "sandwich detection" heuristic for 2-column use/load tables.

Decorative image filtering: Images with alt text under 60 characters matching patterns like "horizontal line", "divider", "decorative" etc. are silently dropped.

6.3 Parser — reference_linker.py

Scans all clause content for cross-references and resolves them against the document ID index.

Entry point:

from parser.reference_linker import link_references

document_dict = link_references(document_dict)

Standard cross-references (added to clause.references[]):

Kind Example Resolution
Sentence / Article / Clause Sentence 4.1.6.5.(1) CL-4-1-6-5
Section / Subsection Section 4.1 SEC-4-1
Table Table 4.1.3.2.-B caption lookup → TBL-N
Figure Figure 4.1.6.5.-A caption lookup → FIG-N

Appendix note references (added to clause.note_refs[]):

Pattern: (See Note A-4.1.6.16.(6).) → resolved via a two-pass note index:
- Pass 1: Clauses whose titles start with A- are indexed
- Pass 2: Embedded text items inside CL-AUTO clauses that start with A- are also indexed

Resolution tries exact match first (A-4.1.6.16.(6)), then base match (A-4.1.6.16).

Important regex fix: The note patterns use \d+(?:\.\d+)* (not [\d\.]+) to correctly parse sub-note identifiers like A-4.1.6.16.(6). The naive [\d\.]+ greedily consumes the dot before .(6) making the sub-clause suffix unreachable.

6.4 Parser — ai_enhancer.py

Optional Claude-powered enhancement. Currently adds semantic column labels to tables.

Entry point:

from parser.ai_enhancer import enhance_document

document_dict = enhance_document(document_dict, use_ai_for_tables=True)

Writes column_semantics[] on each table — a list of {original, semantic} dicts mapping raw header strings to descriptive labels (e.g. "Col 3" → "Factored Resistance (kN/m²)").

Only runs when --ai is passed to main.py.

6.5 Storage — document_store.py

Simple JSON file read/write plus search index builder.

from storage.document_store import save_document, load_document, build_search_index

save_document(document_dict)          # → storage/output/structured_document.json
doc = load_document()                 # loads structured_document.json
index = build_search_index(doc)       # flat list for /search endpoint

build_search_index() returns one entry per clause containing:
- id, type, number, title
- text — all content items concatenated
- snippet — first 200 chars of first text item
- breadcrumb — "Chapter N > M.K > L"
- page — first page of the clause

6.6 API — api/main.py

FastAPI application serving the structured document.

Starting the server:

uvicorn api.main:app --reload --port 8000

The document is loaded once at startup and cached as module globals (_document_cache, _search_index_cache). CORS is configured for http://localhost:8501 (Streamlit).

6.7 Viewer — viewer_streamlit.py

The main user interface. Reads storage/output/structured_document.json directly.

Starting the viewer:

streamlit run viewer_streamlit.py

Deep linking: Append ?clause=CL-4-1-6-5 to the URL to navigate directly to a clause. The viewer reads st.query_params["clause"] on load.

Math rendering: Inline $...$ expressions are rendered as KaTeX by Streamlit's markdown engine. Display equations use st.latex(). HTML tables render math via the KaTeX CDN (v0.16.9) inside an iframe.

Inline note buttons: Text items containing (See Note A-X.X.X.(N).) are split and the note segment is rendered as a clickable button if the note resolves to a clause in this PDF, or as a styled amber badge if it is in an external PDF.

7. Viewer Usage Guide

Browse Mode

  1. Select a Chapter in the left sidebar
  2. Select a Section
  3. Clause content appears in the main area — tabs for ≤6 clauses, a dropdown for more

Navigating references:
- Blue buttons in the clause body = resolved cross-references → click to jump
- Grey badges = unresolved (target is in a different PDF)
- Green buttons = resolved appendix notes → click to jump to appendix clause
- Amber badges = external appendix notes

Search Mode

Type two or more characters to search across all clause titles and body text. Results show a snippet with the matching term in context. Click any result to navigate.

Flagged Issues Mode

While browsing, click the 🚩 Flag an issue expander inside any clause to mark a parsing problem. Choose an issue type and add a note. All flags are saved to storage/output/flagged_issues.json and reviewable in this tab.

Issue types available:
- Missing text, Wrong hierarchy, Table error, Sub-clause split wrong
- Equation wrong, Figure missing, Figure wrong position
- Reference not resolved, Wrong page number, Other

Stats & Raw Mode

Shows:
- Per-chapter clause and section counts
- Reference resolution rate
- Table of all unresolved references per clause
- Download button for the full structured JSON

8. REST API Reference

Base URL: http://localhost:8000

GET /

Health check.

{"status": "ok", "message": "Building Code API is running"}

GET /document

Returns the full document tree. Can be large (1–2 MB).

GET /document/summary

Lightweight navigation tree. Returns chapter/section structure with clause counts but no clause body content. Use this for sidebar navigation in a frontend.

GET /section/{section_id}

Returns a single section with all its clauses.

GET /section/SEC-4-1-6

GET /clause/{clause_id}

Returns a single clause with a _breadcrumb context object.

GET /clause/CL-4-1-6-5

Response includes a _breadcrumb field:

"_breadcrumb": {
    "chapter": {"id": "CH-4", "title": "Structural Design"},
    "section": {"id": "SEC-4-1-6", "title": "Snow Loads"}
}

GET /search?q={term}

Full-text search. Minimum 2 characters. Returns up to 50 results.

GET /search?q=snow+load

{
    "query": "snow load",
    "count": 12,
    "results": [
        {
            "id": "CL-4-1-6-2",
            "number": "4.1.6.2",
            "title": "Specified Snow Load",
            "breadcrumb": "Chapter 4 > 4.1 > 4.1.6",
            "snippet": "...the specified snow load, S, shall be...",
            "page": 38
        }
    ]
}

GET /references/{node_id}

Reverse lookup: which clauses contain a reference pointing to this node.

GET /references/CL-4-1-6-5

9. Testing with Different PDFs

What PDFs Work Well

The parser is optimised for structured technical/regulatory documents such as:

  • Building codes (BCBC, NBC, OBC, IBC, Eurocodes)
  • Engineering standards (CSA, ASTM, ISO)
  • Municipal bylaws with numbered clause hierarchies

Key characteristics of a compatible PDF:
- Numbered sections in a hierarchical pattern (e.g. 4.1.6.5, Clause 3.2.1)
- Consistent heading levels (PDF headings → Datalab SectionHeader blocks)
- HTML tables with <thead> / <tbody> structure
- Equations in MathML or as image+alt-text

Step-by-Step: Testing a New PDF

1. Run the pipeline

python main.py your_new_document.pdf

2. Check the extraction summary

The pipeline prints:

Chapters:    X
Sections:    Y
Clauses:     Z
Tables:      N
References:  NNN found, XX.X% resolved

A reference resolution rate below 80% suggests the document numbering pattern differs from the current regex rules.

3. Open the viewer and spot-check

streamlit run viewer_streamlit.py
  • Open Browse mode and navigate to a known clause
  • Check that the clause hierarchy matches the PDF structure
  • Verify a table is rendered correctly
  • Search for a term you know exists

4. Common issues with different PDFs

Symptom Likely cause What to check
All content in one clause Heading levels not detected Check if Datalab returns SectionHeader blocks; PDF may use images for headings
Low reference resolution Different numbering pattern The PDF may use Clause X.X or Art. X — check REFERENCE_PATTERNS in reference_linker.py
Tables rendering as empty Datalab table HTML format differs Check the raw JSON for table HTML structure
Missing equations Equations as images only Datalab use_llm=true may not have extracted MathML
Wrong clause hierarchy Section numbers have fewer/more parts Adjust RE_SECTION, RE_ARTICLE, RE_SENTENCE regex in structure_parser.py

5. Regenerate after fixing the parser

Since the raw Datalab response is cached, you can fix the parser and re-run at zero cost:

python main.py your_new_document.pdf   # uses cache, reruns parser only

6. Force fresh extraction

Only needed if the PDF itself has changed:

python main.py your_new_document.pdf --force-extract

Adapting Regex Patterns for Different Document Styles

If the new document uses different numbering (e.g. Article 3.2.1 or Section A.1.2), update these patterns in parser/structure_parser.py:

RE_SECTION   = r'^(?:Section\s+)?(\d+\.\d+)\.?\s*(.*)'
RE_ARTICLE   = r'^(\d+\.\d+\.\d+)\.?\s*(.*)'
RE_SENTENCE  = r'^(\d+\.\d+\.\d+\.\d+)\.?\s*(.*)'

And in parser/reference_linker.py:

REFERENCE_PATTERNS = [
    re.compile(
        r'(?P<kind>Sentence|Article|Subsection|Section|Clause)\s+'
        r'(?P<ref>\d+(?:\.\d+)*(?:\.\d+)?(?:\([^)]+\))?)',
        re.IGNORECASE
    ),
    ...
]

10. Troubleshooting

ModuleNotFoundError: No module named 'dotenv'

The virtual environment is not activated, or dependencies are not installed.

# Activate venv first
.venv\Scripts\activate       # Windows
source .venv/bin/activate    # macOS/Linux

pip install -r requirements.txt

[ERROR] DATALAB_API_KEY is not set

Your .env file is missing or the key is the placeholder value. Check:

# Confirm .env exists and has the real key
cat .env

FileNotFoundError: Document not found at storage/output/structured_document.json

You need to run the pipeline before launching the viewer or API:

python main.py path/to/your.pdf

[Datalab] Job did not complete within 300 seconds

The PDF is very large or the Datalab API is slow. Options:
- Retry: run the same command again
- Split the PDF into smaller parts and process each separately

Viewer shows blank content / no clauses

The parser may not have recognised the heading structure. Check the Stats & Raw tab — if Clauses: 0, the headings are likely images (not text) in the PDF, or the numbering pattern doesn't match the regex.

Tables cut off at the bottom

Fixed in the current version (dynamic height estimation + scrolling=True). If it still occurs on a specific table, the table is likely extremely wide — use horizontal scrolling.

References show 0% resolution

The document numbering format differs from the patterns in REFERENCE_PATTERNS. Check the raw JSON (Stats & Raw tab → Download) to see what reference text looks like in the clauses.

11. Known Limitations

Limitation Detail
Single document The pipeline processes one PDF at a time; output overwrites the previous result
Datalab dependency Extraction quality depends on Datalab's LLM-enhanced parser; purely image-based PDFs will produce poor results
English only All regex patterns, heading detection, and reference resolution are English-language specific
Figure images extract_images=false is passed to Datalab (cost/speed tradeoff); figures in the viewer show alt text only unless you change this flag
External references References to clauses in other PDF volumes (e.g. See Note A-3.X.X) cannot be resolved and show as amber badges
No user authentication The FastAPI server has no authentication layer
No multi-document search Search operates within a single processed document

12. Extending the System

Adding a React Frontend

The FastAPI CORS config currently allows http://localhost:8501 (Streamlit). To add a React frontend on port 3000, update api/main.py:

allow_origins=[
    "http://localhost:8501",
    "http://localhost:3000",   # add this
],

Switching to a Database

storage/document_store.py is designed as a swap point. Replace save_document / load_document with PostgreSQL or SQLite calls. The document structure maps naturally to a JSONB column in PostgreSQL.

Multi-Document Support

The current model uses a fixed output path (structured_document.json). To support multiple documents:
1. Add a document registry (a simple JSON index file or a database table)
2. Parameterise load_document(filename) — the signature already accepts a filename argument
3. Update the FastAPI routes to accept a document_id path parameter

Adding More AI Enhancement

The ai_enhancer.py module has stubs for:
- classify_block() — classify ambiguous text blocks
- should_join_fragments() — detect cross-page list continuations
- resolve_ambiguous_reference() — resolve informal references like "the above table"

These are not wired into the main pipeline by default but can be called from enhance_document().

Supporting Other Languages

To support non-English building codes:
1. Update all regex patterns in structure_parser.py and reference_linker.py
2. Update the reference kind matching in _ref_to_id() ("sentence", "article", etc.)
3. Update RE_NOTE and RE_A_TITLE if the appendix note format differs
